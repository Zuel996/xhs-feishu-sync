### Bug 修复：PyInstaller 后端启动 + 配置持久化 ✅

解决 re-run install.bat 后反复出现"无法连接后端"的一系列连环 bug。

---

### Bug 1: `cmd start "" exe` 静默杀死 GUI 子系统进程 🔧

**现象**：install.bat 用 `start "" "%INSTALL_DIR%\xhs-feishu-server.exe"` 启动后端，进程短暂出现后消失，端口 9527 从未绑定。

**根因**：PyInstaller `--noconsole` 模式生成的是 Windows GUI 子系统程序（`console=False`）。`cmd start` 和默认的 `Start-Process` 无法正确启动 GUI 子系统进程，进程启动后立即被杀死。

**修复**：`scripts/install.bat` 第131行改为 PowerShell：
```batch
powershell -NoProfile -Command "Start-Process -FilePath '...' -WorkingDirectory '...' -NoNewWindow"
```
`-NoNewWindow` 是关键参数，不加则进程同样被杀死。

### Bug 2: pystray 阻塞 uvicorn 🔧

**现象**：进程存在（PID 可见，285MB RAM），但端口 9527 从未监听。系统托盘图标正常显示。

**根因**：`pystray.icon.run()` 运行 Windows 消息泵在主线程，uvicorn 在 daemon 线程中 `server.run()` 启动后从未真正开始监听。daemon 线程的 uvicorn 无声失败，无任何错误输出。

**修复**：从 `run_server.py` 中完全移除 pystray + Pillow 依赖。直接在主线程中运行 `uvicorn.run()`。`xhs-feishu-server.spec` 对应移除 pystray/PIL hidden imports。

### Bug 3: 飞书配置重启丢失 🔧

**现象**：`/health` 返回 `feishu_configured: true`，但服务器重启后变回 `false`。用户在插件弹窗中点了"验证并保存"后，配置只存在于 FastAPI 内存变量中。

**修复**：`src/api/server.py` 新增三个函数：
- `_get_config_file_path()` — 解析 `feishu_config.json` 路径（PyInstaller 模式用 exe 目录，开发模式用项目根目录）
- `_save_config_to_file()` — 验证成功后 JSON 持久化
- `_load_config_from_file()` — 启动时自动加载，返回 `FeishuConfig` 或 `None`

全局变量 `_feishu_config` 初始化从 `None` 改为 `_load_config_from_file()`。`/config` 端点成功后调用 `_save_config_to_file()`。

### Bug 4: robocopy 锁定文件静默失败 🔧

**现象**：旧 pystray 版本 exe 在运行时，install.bat 用 robocopy 更新文件。由于 exe 被锁定，新版本未被复制，但 robocopy 报告成功（退出码 ≤ 7）。

**修复**：install.bat 开头 `taskkill /f` 后加入 `timeout /t 2` 等待进程完全退出。robocopy 退出前先确保旧进程已终止。

## 排查中：Content Script 无法拦截 XHS API 请求 ⚠️

### 现象
- 后端运行正常（`/health` 返回 `ok`，飞书已配置）
- 插件弹窗正常工作（可验证飞书、添加账号）
- Content Script 成功注入 XHS 页面（`Content script loaded` 日志可见）
- **但没有 `Profile captured`/`Notes captured` 日志** — XHS API 请求未被拦截

### 初步诊断
在 Content Script 的 fetch 和 XHR hook 中加入了诊断日志，拦截**所有**请求 URL。结果发现：
- Content Script 加载后，hook 中**没有任何 URL 被打印**
- 这说明 XHS 的请求完全绕过了 `window.fetch` 和 `XMLHttpRequest` hook

### 可能原因
1. **XHS 在 Content Script 注入前就保存了 fetch/XHR 引用** — 但我们用的是 `run_at: document_start`，应在所有脚本之前执行
2. **XHS 使用 Service Worker 或其他传输机制** — XHS 页面自己注册了 SW，请求可能经过 SW 拦截而不通过页面的 fetch/XHR
3. **XHS 在主 JS bundle 加载后重新覆盖了 fetch/XHR** — 我们的 hook 被覆盖
4. **XHS 的 API 域名/路径结构与预期完全不同** — 但诊断日志打印所有 URL 也没输出，说明 hook 本身没被调用

### 待继续
- 需要确认 XHS 页面是否使用了 Service Worker 来处理 API 请求
- 如果是，需要在插件的 Service Worker 侧拦截，而非 Content Script
